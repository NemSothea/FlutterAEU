FieldLog is a simple log / record-keeping app that grows **one feature per week**
across 16 weeks. By the end of the course it is a published Flutter app.

This branch/folder is the **Week 1 skeleton (E01.1)** — the foundation, structured
the right way. Every later week extends this codebase rather than fighting it.

---

## What this sample demonstrates

- The **only acceptable project structure**: source code under
  `lib/src/features/<feature>/`, never directly in `lib/main.dart`.
- A 4-line `main.dart` that contains only `runApp()`.
- A separate root widget (`FieldLogApp`) holding app-wide config only.
- A `StatelessWidget` first screen with a single call-to-action.
- A correct `.gitignore` (build artifacts and keystores excluded).
- Two starter widget tests.

## Quick start

> This repository contains the Dart source and configuration only. The platform
> folders (`android/`, `ios/`, etc.) are generated by Flutter and are intentionally
> not committed. Generate them once after cloning.

```bash
# 1. Clone
git clone <repo-url> fieldlog_flutter
cd fieldlog_flutter

# 2. Generate the platform folders around the existing lib/ (one time)
flutter create . --project-name fieldlog_flutter --platforms=android

# 3. Get the packages
flutter pub get

# 4. Run on an emulator or connected phone
flutter run
```

You should see the FieldLog welcome screen with a **Get started** button.
Tapping it shows a SnackBar.

## Project structure

```

├── .gitignore
```

## How to use this sample

This is a **reference**, not something to copy wholesale. The point of E01.1 is
that *you* build this structure yourself, by hand, so the conventions stick.

1. Build your own `fieldlog_flutter` from `flutter create`.
2. Impose the structure shown here.
3. Compare your result against this sample.
4. If yours differs, understand *why* before moving on.

This skeleton was written by hand to model the target structure. Where AI
assistants help most in Week 1: explaining `flutter create` output, debugging
`flutter doctor` issues, and clarifying Dart syntax. Where they hurt: suggesting
outdated package versions in `pubspec.yaml` and proposing a single-file

package versions against [pub.dev](https://pub.dev).
